# chat.log — AI 협업 의사결정 기록

> 전체 대화를 옮긴 것이 아니라, 설계 과정에서 내린 **의사결정의 분기점**만 정리한 기술 회고록이다.
> 각 항목은 `논점 → 검토한 선택지 → 결정과 근거 → AI 협업 방식`으로 구성했다.
> 특히 **AI의 제안을 어디서 수용하고 어디서 의심·수정했는지**에 초점을 둔다.

---

### D1. 서비스 형식: 단일 채널 vs 하이브리드

- **논점**: 뉴스레터(이메일)만 운영할지, 웹사이트만 운영할지.
- **검토**: 뉴스레터는 재방문, 도달에 강하지만 과거 콘텐츠가 휘발됨. 웹은 검색, 수익화에 강하지만 재방문 유도가 약함.
- **결정**: 둘은 상충하는 요구(Push: 기존 구독자 도달 / Pull: 신규 유입, 아카이브·결제)라 **하이브리드**로 동시 충족.
- **협업**: AI가 일반적 구조를 제시 → 나는 이를 "왜 이 구조인가"의 관점으로 다시 정의하고, 제품 구조를 결국 **콘텐츠·구독·발송 세 도메인의 데이터 정합성 문제**로 설정하여 백엔드 설계의 출발점으로 삼았다.

### D2. 구현 범위: 무엇을 만들고 무엇을 뺄까

- **논점**: 2일 안에, "완성도보다 구조화"를 평가하는 과제에서 어디까지 만들까.
- **검토**: 기능을 넓게 만들기 vs 핵심 흐름 하나를 끝까지.
- **결정**: 콘텐츠 & 구독이 교차하는 **프리미엄/일반 구독**을 단일 핵심 규칙으로 잡고 거기 집중. 결제(PG), 실제 이메일 발송 등은 공수 대비 가치가 낮아 **Mock/제외**.
- **협업**: AI가 전체 기능 목록을 제안 -> 나는 핵심/비핵심을 갈라 잘라내고, "왜 빼는가(공수, 비핵심)"를 명확하게 근거를 세움. 제외를 '판단'으로 만든 것이 핵심

### D3. 데이터 모델 교차검증: 두 ERD 산출물 비교

- **논점**: 다른 AI(Gemini)가 만든 더 미니멀한 ERD가 더 적합한가
- **검토**: Gemini안은 plans 테이블이 없고(이진 Free/Premium), 발송 상태가 `sent/opened` 2개뿐. Claude 안은 plans, 발송 큐 상태(`pending`), 인덱스 전략 포함.
- **결정**: Claude 안을 베이스로 채택. 단, **Gemini의 단순화는 수용** — 이진 구독이면 plans는 과설계 소지가 있어 검토 대상으로 남김. 반대로, 내가 명시한 **비동기·배치 발송 요구를 Gemini안은 구조적으로 표현 못 함**(`pending` 상태 부재)을 확인하고(요구사항에 "비동기·배치 발송"이 있어, 미발송 상태(PENDING)를 표현하지 못하기 때문에) 그 모델은 채택하지 않음.
- **협업**: 서로 다른 AI 산출물 2개를 **교차검증**. 더 단순한 쪽을 무비판 수용하지 않고, 내 요구사항을 기준으로 채점. "테이블 수가 많아서가 아니라, 모든 컬럼이 존재 이유를 가질 때 더 나은 설계"라는 판단 기준을 세움.

### D4. PK 전략: bigint vs UUID, 그리고 UUIDv7

- **논점**: 프로덕션에 적합한 기본키는 무엇인가
- **검토**: BIGSERIAL은 작고 빠르나 분산, 비추측에 약함. UUIDv4는 랜덤이라 B-tree 인덱스 단편화 유발.
- **결정**: **UUIDv7(시간순 정렬 UUID)**. 시간 정보가 앞부분에 담겨 인덱스 지역성을 확보하면서 UUID의 장점 유지. PostgreSQL 18 네이티브 `uuidv7()` 사용.
- **협업**: "타임스탬프+UUID를 합치는 게 일반적이냐"는 내 직관을 AI가 UUIDv7로 명명, 검증하고, 최신 PG 지원 여부를 확인. 트레이드오프(생성 시각 노출)도 함께 검토

### D5. 분류 모델: categories vs tags

- **논점**: 카테고리와 태그를 둘 다 둘 필요가 있나
- **결정**: 분류축이 중복되므로 **tags로 일원화하고 categories 제거**.
- **협업**: 기획/설계 단순화 판단은 내가 주도, AI는 그 변경이 인덱스, FK에 미치는 영향을 정리

### D6. 인덱스 전략: 무엇을 왜 색인하나

- **논점**: 한정된 시간에 어디에 인덱스를 둘까.
- **결정**: 가장 자주(프리미엄 권한 확인) 그리고 가장 대량으로(미발송 건 조회) 발생하는 두 질의에 집중
  - 접근 제어: `subscriptions(user_id, end_at) WHERE status='active'` 부분 인덱스 + `UNIQUE(user_id) WHERE active`로 1인 1활성구독 보장 
  - 발송: `WHERE status='pending'` 부분 인덱스(배치 큐) + `UNIQUE(issue_id, user_id)` 멱등성 + `SKIP LOCKED` 동시 처리
- **협업**: AI가 인덱스 개념과 DDL을 제공 -> 나는 그것을 **요구사항(대량/비동기 발송)에 매핑**해, 부분 인덱스, 멱등성, SKIP LOCKED가 왜 이 서비스에 필요한지로 연결

### D7. 스키마 역검증: AI로 내 결과물의 결함을 잡다

- **논점**: ERD 툴로 직접 만든 다이어그램이 무결한가
- **검토·발견**: AI 리뷰로 결함 다수 확인 →
  - 조인 테이블(`contents_tags`, `issue_contents`)에 **PK 누락**(중복 방지 불가)
  - `subscriptions.status` 기본값이 `EXPIRED` -> 신규 구독이 즉시 만료되는 오류
  - `published_at`/`paid_at`이 `NOT NULL`  초안, 실패결제와 모순
  - 타입 오타(`TIEMSTAMP`), 표 이름 오타
- **결정**: 위 항목 전부 교정한 수정본 확정
- **협업**: AI를 맹신하지 않고, **역으로 AI에게 내 산출물을 검증시킴**. 생성->검토->교정 사이클을 통해 무결성 결함을 사람이 최종 책임지고 교정

### AI 활용 방식 요약
 
- 생성보다 '판단'에 활용: 초안 생성은 AI에 맡기되, 채택·삭제·트레이드오프 판단은 직접 내렸다(예: 카테고리 제거, 미니멀 안 기각, 스코프 컷)
- 리뷰어로 활용: 최종으로 만든 산출물을 AI에 검수시켜 결함(PK/FK 누락, 잘못된 기본값)을 조기에 발견/수정했다
- 다른 AI의 더 단순한 안을 요구사항 기준으로 기각한 것과, 내 스키마를 AI로 역검증해 결함을 잡은 부분을 통해 단순히 AI에게만 의존하지 않았음을 확인할 수 있다.
